An AI-powered incident management pipeline that captures UI errors via screenshots and payloads, structures them through a VLM, deduplicates using MongoDB vector search, and conditionally triggers a multi-agent RCA — keeping exactly one canonical record per unique issue.

What it does

Stage 1 — Structurer

User submits screenshot + payload + description. Single VLM call converts it to a clean JSON record with 13 typed fields including embedding-ready text and MongoDB pre-filter metadata.

Stage 2 — Dedup engine

Metadata pre-filter (system, page, error_code, status, last_seen) narrows the candidate set, then $vectorSearch checks cosine similarity. Hit → updateOne to bump count + push evidence. Miss → insertOne.

Stage 3 — RCA agent

Conditional — only fires on new/novel errors. Multi-agent sub-graph: Retrieval → Hypothesis → Validation → Fix suggestion. Writes rca object back to MongoDB.

Stage 4 — Coding agent

Optional extension. Takes RCA hypothesis and produces a patch + auto test-run. Feeds result back into the MongoDB record.

Input to the VLM

System role (static)
  • Identifies as a pipeline analyst, not a chat assistant
  • Explains that output is machine-consumed (MongoDB + embedding)
  • Hard rules: no speculation, exact wording from screenshots, short_summary must be dedup-unique
User role (dynamic — build_structuring_prompt())
  • Injects all ticket fields: requestId, email, routingPath, issueDescription, pageUrl, systemContext, rawPayload hash
  • Screenshot count + per-image instruction
  • 7-step explicit reasoning chain: Read → Examine → Classify → Summarise → Assess → Preliminary → Score

Output schema — 13 required fields

short_summary
≤20 words, dedup-unique, the primary embedding signal
error_type
Enum: ui_crash | api_timeout | 4xx | 5xx | validation | auth | data_missing | layout_broken | ...
full_description
3–5 sentences, exact error message text verbatim
enumerated_report
Array of discrete factual observations from screenshots
system / page / error_code
MongoDB pre-filter fields — extracted here or dedup breaks
severity + severity_weight
Consistent pair: critical↔40, high↔30, medium↔20, low↔10. Seeds priority_score on insertOne
impact_assessment
Who is blocked, is it revenue/auth critical
image_evidence
One entry per screenshot — never empty when images present
preliminary_assessment
Most likely cause, evidence-only, no speculation
confidence_score
0.0–1.0, penalised for missing screenshots / vague descriptions

Pre-filter fields (must be in vector index)

system
Exact match — same subsystem only (billing vs auth are different)
page
URL path exact match or prefix
status
$ne "resolved" — don't match closed issues
last_seen
$gte 30 days ago — rolling window
error_code (optional)
When present, narrow further

On HIT — updateOne ops

$inc: { occurrence_count: 1, priority_score: 10 # auto-escalate }, $set: { last_seen: new Date() }, $push: { evidence: { timestamp: new Date(), page: currentPage, payload_hash: hash, user_id: userId, session_id: sessionId } }

Threshold tuning guide

Start at 0.88. Evaluate on 20–30 historical errors. If false positives (different errors merged): raise to 0.92. If false negatives (same error split): lower to 0.85. One-time calibration, then leave fixed.

numCandidates rule of thumb

Set to 10–20× your limit. For dedup (limit=3), use numCandidates=50. Higher = more accurate ANN but slower. On M0, 50 is safe.

Multi-agent subgraph (MA-RCA pattern)

Retrieval agent
RAG over: codebase, git log, previous RCA docs, runbooks, error logs. Returns ranked relevant context chunks.
Hypothesis agent
Consumes retrieved context + structured error. Generates 2–3 ranked root cause hypotheses with supporting evidence for each.
Validation agent
Dynamically checks hypotheses: call test endpoints, query DB state, run lightweight unit tests. Eliminates unsupported hypotheses.
Synthesis
Top surviving hypothesis + evidence + confidence → write rca{} to MongoDB, update status to "in_rca".

RCA object written to MongoDB

{ rca: { hypothesis: "Null check missing in...", evidence: [ "git blame: changed 2026-03-25", "Unit test XYZ fails on null input" ], fix_suggestion: "Add null guard in processPayment()", fix_code_snippet: "if (!payment) return;", confidence: 0.87, agents_used: ["retrieval", "hypothesis", "validation"], completed_at: ISODate() }, status: "rca_complete" }

When to skip RCA (add to should_do_rca)

  • confidence_score from Stage 1 < 0.5 (not enough info)
  • error_type == "unknown" (can't RCA blindly)
  • A resolved identical error exists (pattern known)

Collection: error_records

Why one collection?
One vector index on M0 free tier. Every record has the same shape. $vectorSearch + pre-filter handles all query patterns. No joins needed.
Why not a separate vector DB?
Your metadata and vector live in the same document. No sync required. $vectorSearch pre-filter runs before ANN — more efficient than a pure vector DB that post-filters.
What about S3?
Raw screenshots + logs go to S3. MongoDB only stores s3_screenshot_url as a reference. Keeps document size small, Mongo fast.
Scaling path
M0 (free, prototype) → Flex (~$8/mo, multiple indexes) → M10 ($57/mo, dedicated + Atlas Search for hybrid). No code changes needed to upgrade.

Key design decisions
Why LangGraph over a single agent?
Vision work isolated in Stage 1. Dedup is text-only and lightweight. RCA only fires conditionally. Full LangSmith tracing per stage. Easy to retry individual nodes.
Why motor (async) for MongoDB?
FastAPI is async. Using synchronous pymongo blocks the event loop. motor is the official async driver and drops in identically.
Why not a dedicated vector DB?
Atlas Vector Search pre-filters before ANN search — more efficient than Pinecone/Qdrant which post-filter. Metadata and vectors live together. One less service to operate.
